# base64-ng 2.0.0 Commit Release Plan

This document converts the 2.0 architecture review into ordered, reviewable
commits. The project does not publish intermediate crate versions from this
plan. Each numbered checkpoint is committed, locally verified, externally
pentested, and accepted before work starts on the next checkpoint. The only
release tag created by this plan is the final signed `v2.0.0` tag.

Nothing listed here is deferred beyond 2.0.0. If a required specification,
toolchain capability, real-hardware result, or security proof is unavailable,
the release waits. An experimental implementation may exist during the commit
sequence, but every item must reach its stated 2.0 exit criteria before GA.

- Candidate release: the core and every workspace companion publish in

- Intermediate checkpoints: numbered Git commits only; no alpha, beta, RC, or

- Active implementation and release toolchain at plan creation: Rust `1.97.1`.
- Minimum supported Rust version at plan creation: Rust `1.90.0`. Commit 4
  re-measures the MSRV, and any change requires its own accepted checkpoint.
- Kani evidence baseline: `cargo-kani 0.67.0` with the documented Rust `1.90.0`

  versions separately from the release toolchain.
- Plan snapshot date: 2026-07-22. Toolchain, dependency, specification, and
  external implementation versions are pinned in the evidence for the commit

The repository remains on the last published package version while the
numbered checkpoints are developed. Commit 54 changes every workspace package
and internal dependency requirement to `2.0.0`; Commit 55 records final
acceptance and is the only commit eligible for the signed `v2.0.0` tag.

## Final Capability And Feature Graph

Cargo features remain additive. The 2.0 design therefore uses this fixed
capability model:

- Ordinary core: default/no-default features; portable; strict scalar with
  optional ordinary SIMD.
- `simd`: additive on supported targets; ordinary encode/decode acceleration
  only.
- `secrets`: additive and broadly portable; compatible with ordinary SIMD but
  implemented through structurally separate scalar secret types and
  best-effort cleanup.
- `checked-backend`: additive on supported targets and compatible with SIMD;
  enables redundant ordinary backend verification/evidence.
- High assurance: an attested build/runtime policy, not a Cargo feature. Its
  `AssuranceContext` enforces algorithm, barrier, and speculation posture and
  rejects SIMD under the required policy. Protected-memory proof remains tied
  to each exact allocation.

The final manifest edges are exact: `std -> alloc`, `stream -> std`,
`checked-backend -> simd`, and `secrets` has no implicit `alloc` edge. A
no-default-features secret array/state remains allocator-free.

`cargo test --all-features` must succeed. The explicit
`base64_ng_require_high_assurance` build policy may intentionally reject a
build that also enables `simd`, but Cargo feature unification alone never
selects high assurance. Enabling `secrets` cannot add `Drop`, wiping, staging,
or other overhead to ordinary state types.

## Final Package Topology

Every published Rust package below uses the workspace MSRV, lockstep `2.0.0`
version, repository unsafe-code policy, and shared sealed consumer traits.
Protocol companions keep parser complexity outside the zero-dependency core.

- `base64-ng`: `no_std`, zero runtime dependencies, optional `alloc`/`std`,
  incremental operation, and a capability-gated scalar secret core.
- Existing derive/serde/bytes/tokio/subtle/sanitization companions: documented
  integration-specific requirements, no hidden core dependency growth, and
  explicit capability-gated secret behavior only.
- `base64-ng-mime`: `no_std + alloc`, optional `std` I/O, incremental, core
  dependency only, and no secret surface.
- `base64-ng-pem`: `no_std + alloc`, optional `std` I/O, incremental, core
  dependency only, and bounded frame integration behind `secrets`.
- `base64-ng-multibase`: `no_std`, optional `alloc` conveniences,
  incremental, core dependency only, and no secret surface.
- `base64-ng-imap`: `no_std`, optional `alloc` conveniences, incremental, core
  dependency only, and no secret surface.
- `base64-ng-password`: `no_std + alloc`, incremental grammar/field transforms,
  no hash implementation, and no password or secret input.
- `base64-ng-openpgp`: `no_std + alloc`, optional `std` I/O, incremental, core
  dependency only, and bounded frame integration behind `secrets`.
- `base64-ng-wasm-loader`: supported JavaScript/npm loader plus scalar/SIMD
  WASM artifacts for ordinary codecs.

All new protocol companions deny unsafe code. Any additional dependency needs
its own review, exact version pin/range policy, license/advisory evidence, and
accepted pentest before Commit 54.

## Gap Analysis Disposition

The 2026-07-22 gap review is incorporated into this executable plan. In
particular, the revised checkpoints now require:

- capability-specific rollback and committed-prefix contracts instead of one
  impossible universal atomicity promise;
- bounded secret frames before plaintext release to public or irreversible
  destinations;
- production backend known-answer tests, quarantine generations, scalar retry
  policy, and high-assurance fail-closed behavior;
- separate best-effort `secrets` capability and attested high-assurance policy;
- explicit const transforms, bounded stack buffers, allocating helpers, and
  validation APIs;
- crate-owned fixed scans for custom secret alphabets;
- extensible public enums, one sealed consumer trait, and independent test
  oracles;
- artifact-level WASM selection and OS/ABI-aware RVV and SVE admission;
- exact protocol versions, scope names, parser resource budgets, and
  independent grammar models; and
- code-size, stack, allocation, pending-memory, and no-secret-reallocation
  evidence in addition to throughput.

The follow-up gap review is also resolved here: Cargo feature unification,
formatter partial writes, secret in-place behavior, exact frame storage,
password-format scope, backend self-test mechanics, assurance generations,
WASM loader ownership, sanitization prevalidation, SIMD proof boundaries,
legacy whitespace, source indexing, companion topology, and async exposure are
all assigned explicit requirements below.

The second follow-up review is resolved as well: staged secret in-place timing,
allocation-specific protection, fast-versus-checked backend writes, panic-free
safe APIs, checked source indexes, content-independent secret progress, stable
stack-limit enforcement, exact feature edges, the JavaScript/WASM contract,
zero-dependency RVV/SVE detection, writer side effects, tail wiping, token
traits, feature-size isolation, and password-record formatting are now explicit.

The third follow-up review fixes the boundary of those claims: secret timing is
fixed-work through the result gate but not through success-only declassification,
panic freedom covers crate-originated behavior rather than hostile callbacks,
memory protection is modeled at logical-allocation and effective-page levels,
and unsafe initialization, JavaScript shared-memory behavior, backend startup,
secret frame ownership, and ordinary-type ABI/auto-trait isolation have exact
checkpoint requirements.

The fourth follow-up review closes the remaining integration ambiguity:
private secret staging is always byte-disjoint from caller-visible storage,
every JavaScript entry point rejects unstable or shared backing storage, and a
versioned cross-crate semantic corpus prevents the core, streaming, Tokio,
bytes, serde, and sanitization surfaces from drifting apart.

The fifth follow-up review makes those rules capability-accurate: corpus
success is byte-identical while failure mutation follows each surface's
declared atomic/rollback/prefix contract, protected-fill is strictly
out-of-place, preflight alias/range failures are non-destructive, and WASM uses
intrinsic typed-array brand/backing checks rather than realm-local
`instanceof` or spoofable properties.

The sixth follow-up review keeps those contracts implementable: genuine
cross-realm `Uint8Array` subclasses are admitted but can be read only through
captured intrinsics, and protected decoding gains a single-allocation typestate
path that validates in place without exposing unvalidated bytes or copying
plaintext to a second protected allocation.

The seventh follow-up review freezes lifecycle and trait boundaries: cleanup
cannot survive deliberate destructor suppression or process termination,
protected-state auto traits are explicit and provider-gated, the direct path is
described as single-allocation rather than zero-copy, and the WASM loader has a
closed callback-free/species-free intrinsic allowlist.

The eighth follow-up review makes the protected cleanup lifecycle atomic and
implementable on the selected MSRV: successful direct finalization has an
explicit RAII-guard handoff with exactly one cleanup owner, conditional `Send`
is provider-proven rather than unconditionally available, and unwind auto
traits are documented as API friction rather than a security boundary.

The ninth follow-up review distinguishes exclusive cleanup ownership from
successful physical teardown. Protected values gain an ordered, fallible
teardown state machine, a reporting consuming close operation, a non-reporting
`Drop` fallback, conservative shared-page accounting, and an explicit
provider-quarantine outcome when cleanup cannot complete.

The tenth follow-up review bounds that quarantine lifecycle. Every provider
admits protected allocations against finite count, logical-byte, effective-page,
registry, and retry-work budgets; reserves an allocation-free quarantine slot
before plaintext exists; and fails closed when provider health or capacity
cannot support another allocation.

The eleventh follow-up review makes wiping a mandatory confidentiality gate in
that lifecycle. Unlock, unprotect, dump re-enablement, disposal, deallocation,
and reuse are unreachable until a complete logical-allocation wipe is verified;
any failed or indeterminate wipe transitions directly to protected quarantine.

## Commit And Pentest Protocol

Every planned commit uses the exact subject prefix shown below, for example:

```text
Commit 1: Freeze the 2.0 scope and evidence contract
```

The workflow for every numbered commit is:

1. Start from the previous accepted checkpoint.
2. Implement only the stated goal and deliverables.
3. Run the commit-specific verification plus `scripts/checks.sh`.
4. Commit with the exact numbered subject.
5. Give the pentester the range from the previous accepted commit through
   `HEAD`.
6. Do not begin the next numbered commit until the pentest reports PASS.

If a pentest finds an issue, remediation commits use the same number with a
letter suffix, such as `Commit 12a: Address incremental decode pentest
findings`. Retest the complete range from the previous accepted checkpoint
through the remediation `HEAD`. Do not rewrite a pushed or reviewed commit.

Each checkpoint must leave the repository buildable and the public security
claims honest. Root `PENTEST.md` remains temporary scratch input and is deleted
after findings are resolved. The final release report records every accepted
checkpoint hash and the final clean retest.

## Global Invariants

These requirements apply to every commit:

- The core crate remains zero-runtime-dependency and `no_std`-first.
- Strict RFC 4648 decoding remains the ordinary default.
- Forgiving, legacy, wrapped, and protocol-specific behavior is explicitly
  named and cannot become the default accidentally.
- Secret APIs use only constant-time-oriented scalar algorithms and mandatory
  cleanup behavior. Ordinary SIMD APIs never masquerade as secret APIs.
- Unsupported backends fall back safely. Backend faults are never reported as
  attacker-controlled malformed input.
- Each API follows its capability-specific commit contract. Atomic rollback is
  promised only where the crate controls rewindable storage; irreversible
  formatters, writers, async sinks, incremental output, and in-place buffers
  report committed progress or a documented post-error state.
- OOM abort, `panic=abort`, process termination, caller-owned borrowed input,
  previously accepted sink output, historical allocator copies, registers,
  caches, swap, and snapshots are outside rollback and cleanup guarantees
  unless a narrower API explicitly proves otherwise. Safe deliberate leaks via
  `mem::forget`, `ManuallyDrop`, `Box::leak`, or ownership cycles can suppress
  destruction; no secret type promises wiping when its destructor does not run.
- At every reachable protected-storage state there is exactly one cleanup
  owner. If consuming close or destruction runs, that owner makes exactly one
  ordered cleanup attempt for the consumed handle. This is not a promise that
  every platform teardown step succeeds. Deliberate destructor suppression,
  abort, OOM abort, and process termination remain excluded; failed teardown
  leaves ownership with an explicit provider-managed quarantine state rather
  than fabricating successful wipe, unlock, accounting, or disposal.
- Protected providers have finite, checked budgets covering active, reserved,
  and quarantined allocations together. Admission fails before plaintext
  materialization when count, logical-byte, effective-page, registry, or
  quarantine capacity is unavailable. Cleanup failure cannot cause unbounded
  resident memory, locked pages, registry growth, or retry work.
- Verified completion of a full logical-allocation wipe is a prerequisite for
  unlock, protection removal, dump re-enablement, disposal, deallocation, or
  reuse. Failed, partial, interrupted, or indeterminate wiping preserves the
  existing effective-page protection and conservative accounting and transfers
  the allocation directly to bounded quarantine.
- Unsafe code remains isolated, inventoried, assembly-reviewed, and covered by
  architecture-specific evidence.
- All workspace crates use lockstep versions at the final release.
- Every new externally sourced protocol behavior is tied to a primary
  specification, exact vectors, and a documented scope boundary.
- Every protocol parser has caller-selectable limits for applicable input,
  output, line, label, header, skipped-byte, block/document, nesting, and
  pre-output work dimensions; defaults are finite and documented.
- No numbered commit is accepted without an external pentest PASS.
- No safe crate implementation originates a panic from malformed input,
  representable exhaustion, arithmetic/index overflow, legal state
  transitions, or resource-limit violations. Panics are limited to documented
  programmer errors and deliberate const-evaluation failures. Panics from
  caller-provided trait implementations or unsafe providers may propagate;
  crate-owned secret state is cleaned during unwinding where supported.

- Define one protected-memory teardown state machine used by explicit close,
  typestate guards, and `Drop`. Enter `Wiping` and wipe the complete logical
  allocation, including initialized bytes and unused capacity. Transition to
  `Wiped` only after every byte write and the required wipe barrier complete
  successfully. Only `Wiped` may attempt page unlock, protection removal, dump

  the provider's declared safety rules. A failed unlock remains conservatively
  accounted as protected and cannot authorize overlapping capabilities to
  claim an unprotected or reusable page.

  bounded retry attempt, and invoke no later teardown hook. Preserve the

  whenever protection status is uncertain. Partial byte progress cannot permit
  partial unlock, unprotect, dump re-enablement, disposal, or reuse.

  attempt as a non-reporting best-effort fallback and discards the report.

  fallible allocation during `Drop`. Consuming `try_close` suppresses the
  wrapper's later `Drop` only after ownership has reached either `Closed` or

  admission-reserved, and quarantined allocation count, logical bytes,
  effective pages, quarantine registry entries, per-entry retry attempts, and
  work per maintenance call. Derive effective-page reservations with checked
  page rounding before allocation; reject any actual protected range that
  exceeds its reservation before plaintext materialization.
- Reserve a generation-tagged quarantine registry slot and its complete
  resource budget before creating or filling protected storage. Return a
  redacted typed `ProtectionResourceExhausted` or `ProviderUnavailable` error
  before decoding if reservation fails. The reserved slot travels with the
  allocation lifecycle and makes later quarantine transfer infallible and
  allocation-free.
- Define provider health as at least `Healthy`, `Degraded`, `Exhausted`, and
  `Shutdown`. Any teardown failure advances the health generation and blocks
  new high-assurance allocations while degraded, exhausted, or shut down.
  Existing operation tokens revalidate that generation and fail closed, while
  cleanup and bounded recovery remain available.
- Make active/reserved/quarantined accounting linearizable, checked against
  all limits as one admission transaction, and generation/ABA-safe. Registry
  slots carry monotonically changing identities independent of allocation
  address reuse; stale capabilities and stale recovery work cannot release,
  reuse, or mutate a newer allocation's accounting.
- Bound recovery to the configured lifetime attempt count per quarantine entry
  and work quota per maintenance call. Successful recovery removes an entry
  only after wipe, protection removal, accounting reconciliation, and disposal
  have genuinely completed. It never reconstructs or returns the consumed
  public handle. Restoring `Healthy` requires an empty quarantine, successful
  provider self-check, explicit re-attestation, and a fresh health generation.

## 2.0.0 Definition Of Done

The release is complete only when all of the following are true:

- Commits 1-55 are accepted in order, and every required pentest reports PASS.
- Conventional all-feature builds succeed; high assurance is an explicit
  build/runtime policy, and feature unification never adds secret behavior to
  ordinary types.
- The ordinary codec has strict, forgiving, incremental, one-shot, in-place,
  formatting, append, chunked, synchronous, and asynchronous APIs with
  documented allocation, rollback, committed-prefix, and post-error contracts.
- Secret operations are explicit, scalar, constant-time-oriented, staged or
  protected as required, redacted, and wiped on every supported unwind/return
  path, with abort, process termination, deliberate destructor suppression,
  caller-owned input, and historical/system-copy exclusions stated explicitly.
- Secret timing claims end at the final result gate. Symbol processing and
  validity accumulation are fixed-work within their documented public-length
  classes; success-only plaintext release and declassification are public and
  may add a length-proportional copy afterward.
- Secret in-place decode validates before mutation; secret streaming releases
  plaintext only from bounded frames or after explicit declassification. Its
  progress and work schedule depend only on public lengths, capacities, and
  padding mode, not byte validity, padding position, or trailing-bit values.
- No safe crate implementation originates a panic for malformed input,
  representable capacity/resource exhaustion, arithmetic overflow, or legal
  state-machine calls. Documented crate panics are limited to programmer or
  const-contract misuse; caller trait/provider panics may propagate while
  crate-owned secret state is cleaned on supported unwinding paths.
- Custom alphabets and wrapping policies are validated values; invalid codec
  specifications cannot be constructed through safe public APIs.
- Every advertised SIMD backend, including RVV and SVE, has real-hardware
  correctness, cleanup, fallback, and performance admission evidence.
- RFC 2045 MIME transfer bodies, RFC 7468 PEM, Base64-family multibase, the RFC
  3501 modified-UTF-7 payload transform, exact admitted password-record
  transforms, and pinned-version OpenPGP armor are complete within their
  named boundaries and have retained interoperability corpora.
- All companion crates use the shared 2.0 core contracts and are versioned as
  the `2.0.0` family.
- Core and companion adapters pass the same versioned semantic conformance
  corpus. Successful semantics agree; failures obey documented per-surface
  mutation and redaction contracts without unintended drift.
- High-assurance protected-memory claims are allocation-specific and
  lifetime-bound. Every distinct plaintext allocation is attested; a direct
  typestate path preserves one stable allocation through validation, while a
  two-allocation path attests staging and destination independently.
  Capabilities cannot survive reallocation, unlock, or degradation.
- Protected teardown has one owner at every reachable point and one ordered
  attempt per consumed handle when close or destruction executes. Cleanup
  failures are redacted, generation-degrading, and conservatively quarantined;
  the release does not claim cleanup when destruction is suppressed or a
  platform teardown step fails.
- Active, reserved, and quarantined protected resources share finite provider
  budgets. Quarantine capacity is reserved before plaintext materialization;
  repeated failures have bounded memory, page, registry, and retry cost; and
  exhausted permanent quarantine shuts the provider down without storage reuse.
- Verified full-range wiping gates every confidentiality-reducing teardown
  action. Wipe failure preserves protection and accounting, invokes no later
  hook, and can end only in bounded full-wipe recovery or permanent protected
  quarantine until process teardown.
- The supported scalar/SIMD WASM loader and artifacts are published as the
  versioned `2.0.0` JavaScript/npm companion.
- Kani, fuzzing, timing/codegen, Miri, sanitizers, cross-platform, MSRV,
  package, SBOM, and clean-room evidence is archived for the exact tag commit.
- Documentation makes no claim broader than the implementation and evidence.
- The signed `v2.0.0` tag is created only after the exact final commit is green
  in required GitHub checks.
